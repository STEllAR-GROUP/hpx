How to write an HPX application ?

structure of HPX source code

 The most flexible way: Supply your own HPX entry point while avoiding to block the main thread

   #include <hpx/hpx_start.hpp> int hpx_main(int argc, char* argv[]) { // Any HPX application logic goes here... return hpx::finalize(); } int main(int argc, char* argv[]) { // Initialize HPX, run hpx_main. hpx::start(argc, argv); // ...Execute other code here... // Wait for hpx::finalize being called. return hpx::stop(); }

   include

    #include <hpx/hpx.hpp>

    #include <hpx/hpx_start.hpp>

   int hpx_main(boost::program_options::variables_map& vm)

    will be invoked as main entry point to HPX application on the console locality only

    will be invoked as first thread of my application

    all HPX API functions can be used form within this function

    different signatures possible:

     int hpx_main()

     int hpx_main(int argc, char* argv[]);

     int hpx_main(boost::program_options::variables_map& vm)

    return hpx::finalize();

     has to be called on one of the HPX localities in order to signal that - all the work has been scheduled - and the runtime system should be stopped after the scheduled work has been executed

   int main(int argc, char* argv[])

    initialize the runtime system

    and pass command line arguments to the program

    you can't use any of the HPX API functions other than hpx::stop from inside the main function

    hpx::start(argc,argv);

     the thread executing hpx::start(...) will NOT block wiating for the runtime to exit, but will return immediately

    return hpx::stop();

     can be used to wait for the HPX runtime system to exit

     should be at least used as last function called in main.

     the value returned from hpx_main will be returned from hpx::stop after the runtime system has stopped.

  Useful for applications where the main thread is used for special operations (eg GUI)

 Balanced use case Supply your own HPX entry point while blocking the main thread

  #include <hpx/hpx.hpp> #include <hpx/hpx_init.hpp> int hpx_main(int argc, char* argv[]) { // Any HPX application logic goes here... return hpx::finalize(); } int main(int argc, char* argv[]) { // Initialize HPX, run hpx_main as the first HPX thread, and // wait for hpx::finalize to be called. return hpx::init(argc, argv); }

   include

    #include <hpx/hpx.hpp>

    #include <hpx/hpx_init.hpp>

   int hpx_main()

    this function will be invoked as main entry point of HPX application on the console locality only

    by default, executed on main locality only. It is possible to force execution on all localities (--hpx:run-hpx-main)

See Also: --hpx:run-hpx-main

    This function will be invoked as first HPX thread of my application

    All HPX API functions can be used form within this function

    different signatures possible:

     int hpx_main()

     int hpx_main(int argc, char* argv[]);

     int hpx_main(boost::program_options::variables_map& vm)

    content

     do whatever you have to do ...

     return hpx::finalize();

      tells hpx to shutdown

       waits for all activity (system-wide) to cease

       releases main threads waiting inside hpx::init()

      can be called on any locality

      signals that all the work has been scheduled and the runtime system should be stopped after the scheduled work has been executed

      first invocation triggers the exit

   int main(int argc, char* argv[])

    initializes HPX runs HPX_main as the first HPX thread waits for hpx::finalize to be called

    // at the end: return hpx::init(argc, argv);

     the thread executing the function hpx::init will block waiting for the runtime system to exit

     The value returned from hpx_main will be returnedby hpx::init after the runtime system has stopped

     other possibilities:

      hpx::init(desc_commandline, argc, argv)

       in hpx_init.hpp, .cpp and hpx_init_impl.hpp (with various overloads)

       starts the runtime system (ie initializes HPX)

       by default: invokes hpx_main() as first thread

  allows to - decide which version of hpx::init to call - pass additional configuration parameters while initializing the HPX runtime system

  with command line options:

   #include <hpx/hpx.hpp> #include <hpx/hpx_init.hpp> #include <boost/format.hpp> int hpx_main(boost::program_options::variables_map& vm) { // extract cmd line arg std::uint64_t option_value = vm["my_option"].as<std::uint64_t>(); // body of main return hpx::finalize(); } int main(int argc, char* argv[]) { boost::program_options::options_description desc_commandline("Usage: " HPX_APPLICATION_STRING " [options]"); desc_commandline.add_options() //! how to use: --my_option 10 ("my_option", boost::program_options::value<std::uint64_t>()->default_value(0), "sets blahblah for blahblah"); return hpx::init(desc_commandline, argc, argv); }

 Minimally invasive: re-use main() as main HPX entry point (with a dirty trick)

   #include <hpx/hpx_main.hpp> int main(int argc, char* argv[]) { return 0; }

   include

    #include <hpx/hpx.hpp>

    #include <hpx/hpx_main.hpp>

   int main(int argc, char* argv[])

    then HPX will invoke the main() as its first HPX thread.

    the runtime system will be initialized behind the scenes before main() is executed and will automatically stop after main() has returned.

    All HPX API functions can be used from within this function

    could also expose the simple signature int main()

   no need for an hpx::init() nor an int hpx_main()

  To achieve this seamless integration, the header file hpx/hpx_main.hpp defines a macro #define main hpx_startup::user_main which could result in unexpected behavior.

how to analyze the performance?

 performance counters

  what is it?

   Performance Counters in HPX are used to provide information as to how well the runtime system or an application is performing. Performance Counters are HPX parallel processes which expose a predefined interface. HPX exposes special API functions that allow one to create, manage, read the counter data, and release instances of Performance Counters. Performance Counter instances are accessed by name, and these names have a predefined structure which is described in the section Performance Counter Names. The advantage of this is that any Performance Counter can be accessed remotely (from a different locality) or locally (from the same locality). Moreover, since all counters expose their data using the same API, any code consuming counter data can be utilized to access arbitrary system information with minimal effort.

  command line options

See Also: performance

   --hpx:print-counter

    print the specified performance counter either repeatedly and/or at the times specified by --hpx:print-counter-at (see also option --hpx:print-counter-interval)

    --hpx:print-counter /threads{locality#0/worker-thread#0}/idle-rate

   --hpx:print-counter-reset

    print the specified performance counter either repeatedly and/or at the times specified by --hpx:print-counter-at, reset the counter after the value is queried. (see also option --hpx:print-counter-interval)

   --hpx:print-counter-interval

    print the performance counter(s) specified with --hpx:print-counter repeatedly after the time interval (specified in milliseconds), (default: 0, which means print once at shutdown)

   --hpx:print-counter-destination

    print the performance counter(s) specified with --hpx:print-counter to the given file (default: console)

   --hpx:list-counters

See Also: what names are available? (get them with )

    list the names of all registered performance counters, possible values: minimal (prints counter name skeletons), full (prints all available counter names)

   --hpx:list-counter-infos

See Also: what names are available? (get their description with)

    list the description of all registered performance counters, possible values: minimal (prints info for counter name skeletons), 'full' (prints all available counter infos)

   --hpx:print-counter-format

    print the performance counter(s) specified with --hpx:print-counter, possible formats in csv format with header or without any header (see option --hpx:no-csv-header), possible values: csv (prints counter values in CSV format with full names as header), csv-short (prints counter values in CSV format with shortnames provided with --hpx:print-counter as --hpx:print-counter shortname,full-countername)

   --hpx:no-csv-header

    print the performance counter(s) specified with --hpx:print-counter and csv or csv-short format specified with --hpx:print-counter-format without header

   --hpx:printer-counter-at arg

    print the performance counter(s) specified with --hpx:print-counter (or --hpx:print-counter-reset) at the given point in time, possible argument values: startup, shutdown (default), noshutdown

   --hpx:reset-counters

    reset all performance counter(s) specified with --hpx:print-counter after they have been evaluated

   --hpx:print-counter-at

    print the performance counter(s) specified with --hpx:print-counter (or --hpx:print-counter-reset) at the given point in time, possible argument are values: 'startup', 'shutdown' (default), 'noshutdown'.

   --hpx:print-counters-locally

    Each locality prints only its own local counters. If this is used with --hpx-print-counter-destination=<file>, the code will append a ".<locality_id>"` to the file name in order to avoid clashes between localities.

  names

   All Performance Counter instances have a name uniquely identifying this instance. This name can be used to access the counter, retrieve all related meta data, and to query the counter data

   general form:

    /objectname{full_instancename}/countername@parameters

    where full instance name can be another counter name or a string formatted as

    parentinstancename#parentindex/instancename#instanceindex

    if an index is not specified, HPX will assume a default of -1
